# 모니터링 위드 에이전트 데모 청사진

## 1. 프로젝트 맥락

- 목표: 내부 이커머스 더미 서비스를 대상으로 표준 모니터링과 에이전트 기반 장애 요약 가치를 증명하는 데모를 만든다.
- 주요 사용자: 구매자, 판매자, 관리자.
- 고정 관측 스택: `Grafana + Loki + Prometheus`.
- 고정 알림 경로: `Grafana Unified Alerting`이 `Prometheus`와 `Loki`를 조회해 알림을 평가한다.
- 1차 제외 범위: 실 PG 연동, 배송, 외부 검색 API, Gemini 같은 외부 AI API, Tempo/Mimir.
- 1차 핵심 흐름: 상품 조회/검색 -> 장바구니 -> 주문 생성 -> 모의 결제 시도/결과 -> 관리자 관측.

## 2. 이번 문서에서 확정한 결정

- 데모는 완성형 쇼핑몰이 아니라 관측 가능한 핵심 구매 퍼널을 보여주는 실험장으로 만든다.
- 1주차 구조는 `모듈형 모놀리식`으로 고정하고, 앱 프로세스 안에 요약 에이전트 모듈을 둔다.
- 1차 에이전트는 외부 호출 없는 `읽기 전용 규칙 기반 요약기`로 고정한다.
- 도메인 이벤트는 별도 이벤트 버스 없이 `구조화 로그`로 남긴다.
- 메트릭 라벨은 저카디널리티만 허용하고, `product_id`, `cart_id`, `order_id`, `payment_id`는 로그에서만 사용한다.

## 3. MVP 범위

### 포함

- 상품 목록 조회
- 상품 상세 조회
- 내부 검색
- 내부 추천 플레이스홀더
- 장바구니 생성/수정
- 주문 생성
- 모의 결제 시작/성공/실패
- 관리자용 대시보드
- 인시던트 요약 카드

### 제외

- 실 결제 승인
- 배송/운송 상태
- 외부 추천/검색 서비스 연동
- 광범위한 백오피스 기능
- ERD 전체 도메인 구현

## 4. 권장 아키텍처

### 서비스 형태

- 빠른 구현과 데모 재현성을 위해 모듈형 모놀리스를 사용한다.
- 내부 모듈 예시:
  - `catalog`
  - `search`
  - `recommendation`
  - `cart`
  - `order`
  - `payment`
  - `monitoring`

### 런타임 컴포넌트

- `app`: 더미 이커머스 API와 간단한 UI
- `prometheus`: 메트릭 수집과 저장
- `grafana`: 대시보드와 통합 알림
- `loki`: 구조화 로그 저장과 조회
- `agent-module`: 앱 내부에서 동작하는 읽기 전용 요약 모듈

### 원칙

- 많은 기능을 얕게 만드는 대신, 좁은 퍼널을 깊게 계측한다.
- ERD는 미래 확장 기준으로 참고하되 1차 구현 범위를 강제하지 않는다.
- 1주차에는 `/metrics` 엔드포인트 하나와 구조화 로그 체계를 우선 완성한다.

## 5. 핵심 사용자 흐름

1. 구매자가 상품 목록 또는 검색 결과를 본다.
2. 구매자가 상품 상세를 본다.
3. 구매자가 장바구니에 상품을 담는다.
4. 구매자가 수량이나 옵션을 수정한다.
5. 구매자가 주문을 생성한다.
6. 구매자가 모의 결제를 시작한다.
7. 시스템이 결제 성공 또는 실패 상태를 남긴다.
8. 관리자가 대시보드와 인시던트 요약을 확인한다.

## 6. 관측 모델

### 메트릭 설계 원칙

- 요청 기반 API는 `RED`(Rate, Errors, Duration)로 본다.
- 인프라는 `USE`(Utilization, Saturation, Errors)로 본다.
- 메트릭 라벨은 `service`, `handler`, `method`, `status_code` 정도로 제한한다.
- 엔티티 식별자는 로그로만 남기고 메트릭 라벨에는 넣지 않는다.

### 기본 메트릭

- 시스템 헬스
  - CPU 사용률
  - 메모리 사용률
  - 재시작 횟수
  - 서비스 가용성
- API 헬스
  - 요청 수
  - 오류 수
  - p95 지연 시간
  - 엔드포인트 성공률
- 비즈니스 퍼널
  - 상품 상세 조회 수
  - 검색 요청 수
  - 장바구니 담기 수
  - 주문 생성 수
  - 결제 시도 수
  - 결제 성공률
  - 결제 실패율

### 구조화 로그

- 모든 핵심 요청은 JSON 로그를 남긴다.
- 도메인 이벤트는 `event_name` 필드를 가진 구조화 로그 레코드로 기록한다.
- 상관관계 추적은 로그 기준으로 한다.
- 권장 필드:
  - `timestamp`
  - `level`
  - `service`
  - `environment`
  - `request_id`
  - `user_role`
  - `endpoint`
  - `product_id`
  - `cart_id`
  - `order_id`
  - `payment_id`
  - `result`
  - `error_code`

### 도메인 이벤트

- `product.list_viewed`
- `product.detail_viewed`
- `search.executed`
- `recommendation.shown`
- `cart.created`
- `cart.item_added`
- `cart.item_updated`
- `order.created`
- `payment.started`
- `payment.succeeded`
- `payment.failed`

## 7. 대시보드 구성

### 시스템 헬스 대시보드

- CPU, 메모리, uptime, 재시작 수
- 컴포넌트별 가용성
- 최근 경고/오류 로그

### API 헬스 대시보드

- 엔드포인트별 요청량
- 엔드포인트별 오류율
- 검색, 장바구니, 주문, 결제 API 지연 시간
- 최근 실패 요청과 상관관계 ID

### 비즈니스 퍼널 대시보드

- 목록 -> 상세 -> 장바구니 -> 주문 -> 결제 시도 -> 결제 성공
- 단계별 이탈률
- 엔드포인트/역할 기준 실패 분포
- 필요 시 `product_id`는 로그 드릴다운으로 확인

### 인시던트 요약 대시보드

- 활성 알림
- 상위 실패 유형
- 에이전트 요약 카드
- 메트릭/로그로 이동하는 링크

## 8. 알림 규칙

- 알림 평가는 `Grafana Unified Alerting`만 사용한다.
- Prometheus는 수집/저장과 PromQL 조회 기준으로 사용한다.
- Loki는 LogQL 기반 패턴 탐지에 사용한다.

### 시작 임계치

| 구분 | 신호 | 시작 기준 | 심각도 | 비고 |
|---|---|---:|---|---|
| 인프라 | CPU 사용률 | `> 80%` 5분 | 중간 | 합의된 시작값 |
| 인프라 | 메모리 사용률 | `> 85%` 5분 | 높음 | 합의된 시작값 |
| API | 5xx 오류율 | `> 3%` 5분 | 높음 | 핵심 엔드포인트 기준 |
| API | p95 지연 시간 | `> 1s` 5분 | 중간 | 엔드포인트별 조정 |
| 비즈니스 | 결제 실패율 | `> 10%` 5분 | 높음 | 모의 결제여도 중요 |
| 비즈니스 | 주문 생성 실패 | `5분 내 5건 초과` | 높음 | 데모 인시던트 핵심 |
| 비즈니스 | 장바구니->주문 이탈 급증 | 기준선 대비 급증 | 중간 | 퍼널 이상 탐지 |
| 로그 | 동일 오류 코드 급증 | `5분 내 burst` | 중간 | LogQL 패턴 기준 |

## 9. 에이전트 범위

### 구현 제약

- 1차 에이전트는 네트워크를 호출하지 않는다.
- 권장 구현은 `PromQL + LogQL` 결과를 읽는 규칙 기반 요약기다.
- 향후 로컬 모델을 붙이더라도 읽기 전용 원칙은 유지한다.

### 해야 하는 일

- 메트릭, 로그, 이벤트를 읽는다.
- 퍼널 이상 징후를 요약한다.
- 어디서 문제가 생겼는지와 다음 확인 항목을 제안한다.
- 기술 신호와 비즈니스 영향의 연결을 보여준다.

### 하지 말아야 할 일

- 외부 API 호출
- 앱 상태 변경
- 자동 복구 실행
- 근거 없는 원인 단정

### 출력 형식

- `문제`: 무엇이 이상한가
- `영향`: 어떤 사용자 흐름이 영향을 받는가
- `근거`: 관련 메트릭/로그/이벤트
- `가능성 높은 원인`: 우선순위 가설
- `다음 확인`: 운영자가 바로 볼 항목

## 10. 1주차 완료 기준

- 핵심 엔드포인트 5개 이상 계측 완료
- 기준 메트릭 10개 이상 노출
- Grafana/Loki용 로그 쿼리 5개 이상 합의
- 초기 알림 규칙 3개 이상 동작
- 시스템/API/비즈니스 대시보드 3개 초안 완성
- 인시던트 요약 예시 1개 이상 시연 가능

## 13. 열린 질문

- 1차 데모에서 구매자 화면만 필요한가, 관리자 화면까지 꼭 필요한가?
- 검색은 단순 내부 필터 수준으로 충분한가, 아니면 검색 품질 이슈를 보여줄 정도의 룰이 필요한가?
- 첫 에이전트 요약은 임계치 기반만 쓸지, 로그 패턴 상관분석까지 넣을지?
