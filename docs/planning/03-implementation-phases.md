# 구현 단계 계획

## 목표

- 데모를 한 번에 크게 만들지 않고, `서비스 구축 -> 표준 모니터링 설계 -> 기준표 확정 -> 시나리오 검증 -> 에이전트 요약 -> 데모 패키징` 순서로 나눈다.

## Phase 0. 계획 기준선 확정

### 목적

- 범위, 아키텍처, 신호 이름, 협업 방식을 고정한다.

### 산출물

- `docs/planning/01-demo-blueprint.md`
- `docs/planning/02-sli-alert-matrix.md`
- 핵심 흐름과 제외 범위 합의
- 알림 경로와 에이전트 제약 합의

### 종료 조건

- 외부 API, 실 PG, 배송이 범위 밖임을 모두 동의한다.
- 팀이 같은 사용자 흐름을 보고 있다고 말할 수 있다.
- SLI/장애/알림/우선순위 표의 기본 틀이 만들어진다.

## Phase 1. 서비스 구축

### 목적

- 실제 모니터링 신호를 만들 수 있는 최소 이커머스 서비스를 구현한다.

### 범위

- 상품 목록
- 상품 상세
- 내부 검색
- 추천 플레이스홀더
- 장바구니 담기/수정
- 주문 생성
- 모의 결제 시작/결과

### 산출물

- 모듈형 모놀리식 앱
- 더미 상품/주문 데이터
- 핵심 API 계약
- 실패를 재현하는 토글 또는 플래그

### 종료 조건

- 행복 경로가 끝까지 동작한다.
- 핵심 엔드포인트 5개 이상이 안정적으로 응답한다.
- 의도적으로 실패 시나리오를 만들 수 있다.

## Phase 2. 표준 모니터링 설계

### 목적

- 서비스 위에 표준 관측 구조를 올린다.

### 범위

- `Prometheus` 메트릭 수집
- `Loki` 구조화 로그 수집
- `Grafana` 대시보드 및 통합 알림
- 메트릭 이름/라벨 규칙
- 로그 스키마와 상관관계 ID 규칙

### 산출물

- `/metrics` 엔드포인트
- API용 RED 메트릭
- 인프라용 USE 메트릭
- 구조화 JSON 로그
- Grafana 데이터소스와 대시보드 초안
- Grafana Unified Alerting 경로

### 종료 조건

- Prometheus에서 메트릭을 질의할 수 있다.
- Loki에서 로그를 질의할 수 있다.
- Grafana에서 시스템/API/비즈니스 퍼널 대시보드를 볼 수 있다.

## Phase 3. SLI / SLO / SLA 및 운영 기준 확정

### 목적

- 수집한 신호를 운영 기준으로 바꾼다.

### 범위

- SLI 정의
- SLO 목표값
- 내부 참고용 SLA 초안
- 장애 유형 분류
- 알림 정책
- 운영 우선순위

### 산출물

- 기준표 문서
- 초기 알림 규칙 3~8개
- 심각도 체계: 낮음/중간/높음/치명
- 각 핵심 장애에 대한 첫 확인 항목

### 종료 조건

- 각 핵심 흐름마다 최소 1개 이상의 SLI가 있다.
- 치명/높음 알림은 대응 액션이 분명하다.
- 돈을 지키는 지표와 돈을 버는 지표가 둘 다 정의된다.

## Phase 4. 시나리오 주입 및 검증

### 목적

- 실제로 장애를 넣어보고 설계가 먹히는지 확인한다.

### 범위

- 검색 지연 증가
- 주문 생성 실패
- 결제 실패 급증
- 동일 오류 로그 반복
- 퍼널 전환 급락

### 산출물

- 시나리오 스크립트 또는 플래그
- 대시보드 캡처 또는 녹화
- 알림/로그/요약 카드 검증 기록

### 종료 조건

- 상위 3개 장애 유형을 반복 재현할 수 있다.
- 알림과 대시보드가 의도대로 반응한다.
- 오탐이나 누락이 문서화된다.

## Phase 5. 에이전트 요약 계층

### 목적

- 기존 신호를 읽어 운영자에게 읽기 쉬운 설명으로 바꾼다.

### 범위

- 규칙 기반 이상 징후 요약
- PromQL / LogQL 질의 템플릿
- 운영자용 요약 포맷
- 관리자 화면 또는 Grafana 패널 연결

### 산출물

- 로컬 읽기 전용 요약 모듈
- `문제 / 영향 / 근거 / 가설 / 다음 확인` 템플릿
- 최소 3개 시나리오 요약 예시

### 종료 조건

- 요약이 실제 신호와 맞는다.
- 외부 API를 호출하지 않는다.
- 운영자가 요약에서 로그/메트릭으로 바로 이동할 수 있다.

## Phase 6. 데모 패키징

### 목적

- 누구나 로컬에서 같은 데모를 재현할 수 있게 묶는다.

### 범위

- docker compose 실행
- 시드 데이터
- 빠른 시작 가이드
- 발표 흐름 정리

### 산출물

- 한 번에 뜨는 로컬 실행 구성
- 시드 스크립트
- 정상 시나리오와 장애 시나리오 데모 스크립트
- 최종 캡처 또는 walkthrough 문서

### 종료 조건

- 새 팀원도 로컬에서 데모를 실행할 수 있다.
- 정상 흐름 1개와 장애 흐름 1개 이상을 재현할 수 있다.
- 모니터링과 에이전트 출력이 리뷰 가능한 수준으로 안정화된다.

## 권장 역할 분담

| 트랙 | 권장 담당 |
|---|---|
| 서비스 | 앱 흐름, API 계약, 더미 데이터, 실패 토글 |
| 모니터링 | Prometheus, Grafana, Loki, 라벨 규칙, 대시보드, 알림 |
| 분석 | 장애 분류, 에이전트 요약 포맷, 시나리오 검증 |

## 진행 순서 원칙

1. Phase 0 없이 대시보드부터 만들지 않는다.
2. Phase 1 없이 알림부터 튜닝하지 않는다.
3. Phase 2 없이 에이전트 요약 품질을 논하지 않는다.
4. Phase 4 없이 모니터링이 완성됐다고 말하지 않는다.
5. 구현이 밀리면 기능보다 관측 품질을 우선 지킨다.

## 초기에 하지 말아야 할 것

- 실 PG 연동
- 외부 검색/추천 API 연동
- 과도한 마이크로서비스 분리
- 고카디널리티 메트릭 추가
- 자동 복구 기능 선행 구현
