# 장애 시나리오 스토리 예시

버튼 클릭으로 장애를 의도적으로 유발하고, 모니터링 지표 변화를 관찰하기 위한 시나리오 문서입니다.

## 시나리오 A: 강제 에러 (HTTP 500)

- 현실 스토리: 결제 서비스가 외부 PG 응답 포맷 변경을 반영하지 못해 예외 발생
- 버튼 의미: 결제 API 예외 모드 ON
- 모니터링에서 볼 것: 5xx 급증, 에러율 상승, 실패 로그 증가
- 핵심 메시지: 외부 의존성 변경은 작은 차이로도 대규모 장애를 만든다

## 시나리오 B: 지연 주입 (Latency 증가)

- 현실 스토리: DB 인덱스 누락 + 트래픽 급증으로 쿼리 응답이 느려짐
- 버튼 의미: DB 응답 지연 모드 ON
- 모니터링에서 볼 것: p95/p99 응답시간 급등, 타임아웃/재시도 증가
- 핵심 메시지: 에러가 없어도 느려지면 사용자 체감은 이미 장애

## 시나리오 C: 리소스 압박 (CPU/메모리 증가)

- 현실 스토리: 신규 배치 작업이 피크 시간대와 겹쳐 애플리케이션 자원 고갈
- 버튼 의미: 백그라운드 과부하 모드 ON
- 모니터링에서 볼 것: CPU/메모리 급등, 처리량 감소, GC/스로틀링 증가
- 핵심 메시지: 기능 자체는 정상이어도 자원 경쟁으로 전체 서비스 품질이 무너짐

## 시나리오 D: 부분 장애 (특정 기능만 실패)

- 현실 스토리: 캐시 서버 장애로 추천/검색만 느려지고 핵심 주문은 동작
- 버튼 의미: 추천 서비스 장애 모드 ON
- 모니터링에서 볼 것: 특정 엔드포인트 에러율만 상승, 전체 에러율은 낮게 보일 수도 있음
- 핵심 메시지: 전체 평균 지표만 보면 놓치는 장애가 있다

## 시나리오 E: 장애 전파 (Cascading Failure)

- 현실 스토리: 한 다운스트림 장애가 재시도 폭증을 만들고, 결국 전체 API까지 영향
- 버튼 의미: 외부 API timeout 모드 ON
- 모니터링에서 볼 것: 대기열 증가, 스레드풀 고갈, 연쇄 타임아웃
- 핵심 메시지: 복구보다 중요한 건 전파 차단(타임아웃/서킷브레이커)
