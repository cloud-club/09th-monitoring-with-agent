# 디스커버리 플랜

**작성일**: 2026-03-22
**제품 단계**: 신규 내부 데모 제품
**디스커버리 질문**: 가장 작은 내부 이커머스 데모로도 표준 모니터링과 에이전트 기반 장애 요약의 가치를 충분히 보여줄 수 있는가?

## 1. 디스커버리 전제

- 지금 검증하려는 것은 공개 쇼핑몰이 아니라 `모니터링 데모 제품`이다.
- 핵심은 더미 서비스 자체보다 `관측 가능성`과 `설명 가능성`이다.
- 고정 제약:
  - 외부 API 없음
  - 실 PG 없음
  - 배송 없음
  - 스택은 `Grafana + Loki + Prometheus`
  - 1차 에이전트는 로컬, 읽기 전용

## 2. 아이디어 발산

| 번호 | 아이디어 | 의미 |
|---|---|---|

| 5 | 규칙 기반 장애 요약기 | 외부 AI 없이도 에이전트 가치를 보여준다 |
| 6 | 검색 품질 계측 | 결제 외의 현실적 장애 축을 만든다 |

| 8 | 인시던트 시나리오 팩 | 데모를 반복 검증 가능하게 만든다 |
| 9 | 관리자 리뷰 흐름 | 운영자 관점의 사용성을 만든다 |
| 10 | 협업용 운영 기준표 | 구현물을 서로 비교 가능한 상태로 만든다 |

## 6. 가정 우선순위화

### 가장 위험한 도약 가정

- `A1`: 퍼널이 너무 단순하면 데모가 장난감처럼 보일 수 있다.
- `A4`: 요약이 쓸모없으면 에이전트 계층이 불필요해 보인다.
- `A8`: 단계 통제가 실패하면 쇼핑몰 기능 확장으로 범위가 무너진다.

### 높은 영향 / 높은 위험

- 최소 퍼널로 가치 입증 가능
- 규칙 기반 요약의 유용성
- 단계 기반 구현의 범위 통제 가능성

### 높은 영향 / 낮은 위험

- 현재 관측 스택만으로 1차 데모 가능
- 텔레메트리 계약이 팀 정렬에 도움을 줌

## 8. 실험 상세

### E1. 데모 스토리보드 리뷰

- 가설: 현재 퍼널만으로도 데모 목적이 충분히 전달된다.
- 측정: 리뷰어의 “반드시 추가해야 할 흐름” 요구 건수
- 판단: 반복적으로 같은 누락 흐름이 나오면 Phase 1 또는 4에서 정확히 1개만 추가한다.

### E2. 시드 장애 팩

- 가설: 재현 가능한 장애가 있어야 모니터링 설계가 설득력을 갖는다.
- 측정: 경보 발생, 로그 조회, 대시보드 확인, 요약 생성의 4단계 통과 여부
- 판단: 한 단계라도 빠지면 기능 확장보다 계측 보완을 먼저 한다.

### E3. 규칙 기반 요약 시험

- 가설: 외부 AI 없이도 운영자에게 도움이 되는 설명을 만들 수 있다.
- 측정: 요약 카드만 보고 문제, 영향, 근거, 다음 확인을 말할 수 있는가
- 판단: 설명이 모호하면 로컬 LLM보다 먼저 근거 포맷을 개선한다.

## 9. 디스커버리 타임라인

- 1주차: 범위 고정, 텔레메트리 스키마, 시드 장애, 서비스 골격
- 2주차: 대시보드, 알림, 규칙 기반 요약
- 3주차: 검증 리뷰, 데모 패키징, 로컬 모델 확장 여부 판단

## 10. 의사결정 기준

- 시드 장애가 end-to-end로 보이면 데모 패키징으로 진행한다.
- 대시보드는 되지만 요약이 약하면 데모는 유지하고 요약 고도화는 후속으로 미룬다.
- 퍼널이 너무 얇다고 평가되면, 모니터링 가치에 직접 연결되는 인접 흐름 1개만 추가한다.
- 일정이 밀리면 추천 품질보다 관측 품질을 우선 지킨다.
